# Agent 说明

## 项目目标

这是一个基于代理服务器的 Codex 账户管理工具，核心目标是：

- 通过代理层替换 `accessToken`，实现多账号无感切换
- 自动同步额度、类型、刷新时间、流量等状态到内存和 UI
- 通过自动负载策略，优先使用额度更充足的账号
- UI 尽量只展示用户关心的信息，内部同步和切换尽量自动化

## 环境约束

- 优先使用 `conda` 环境运行
- 环境名称：`codex_session`
- Python 版本：`3.11`
- 启动或执行任务前，先检查该环境是否存在
- 如果 `codex_session` 不存在，不要直接创建，先询问用户是否需要创建
- 如果用户确认需要创建，则使用 `conda` 创建 `codex_session`，并确保 Python 版本为 `3.11` 或更高
- 依赖统一维护在 `requirement.txt`，新增或修改第三方依赖时优先更新该文件

## 代码分层

- `main.py` 只作为程序入口，不承载业务逻辑和界面实现
- UI 代码统一放在 `app/ui/`
- 业务逻辑、数据读取、状态更新统一放在 `app/services/`
- 数据模型、DTO、配置结构统一放在 `app/models/`
- 通用工具函数统一放在 `app/utils/`
- 新功能优先按“入口、UI、服务、模型、工具”分层组织，避免把所有逻辑堆到一个文件里

## 关键业务规则

- `~/.codex/auth.json` 是源授权文件，`auth/` 目录是本地镜像和管理目录
- `AuthSyncService` 负责源文件和管理目录的同步，UI 不直接读写文件
- `AuthUsageService` 负责额度刷新、计划类型和刷新时间缓存
- `ProxyLoggerAddon` 负责代理空闲检测、请求头改写和使用回报
- 自动负载默认开启
- 代理空闲超过 `5` 秒时，重新选择一次负载目标
- 负载目标按 `5小时` 剩余额度最高优先，若相同再按其他稳定字段做次级排序
- 代理层对所有请求都可替换 `Bearer accessToken`，不限制单一 URL
- 自动负载开启时，替换后的目标 token 也要计入流量
- 自动负载关闭时，保留原始 token 的计数逻辑
- 流量默认值是 `0`
- `quota_refresh_time` 必须来自接口返回的 `rate_limit.*.reset_at`，不是本地打点时间
- `quota` 展示使用接口返回的额度摘要，`plan_type` 展示计划类型

## UI 约束

- 授权表里隐藏 `刷新令牌` 列，不把它作为主展示项
- 表格里保留 `当前` 和 `负载` 标记列，分别表示当前授权和自动负载目标
- `账户id`、`令牌刷新时间`、`额度刷新时间`、`额度`、`类型`、`流量` 是主要展示列
- 表格列宽优先保证 `账户id`、`额度`、`流量` 可读，避免明显留白
- `额度刷新时间` 没有值时显示空字符串
- Tooltip 可以展示更完整的调试信息，但主表格保持简洁
- 刷新类操作必须异步执行，按钮进入禁用状态，完成或失败后必须恢复
- 状态变化要通过回调刷新 UI，不要依赖用户手动刷新

## 修改原则

- 只做最小改动，避免整文件重写
- 禁止随意删除用户已有改动
- 修改文件必须使用 UTF-8 无 BOM
- 修改后需要回读并确认中文没有乱码
- 涉及数据结构或表格列变化时，要同步检查 UI 插入值和列顺序是否一致
- 涉及刷新时间、流量、负载目标时，要检查内存缓存、同步服务和 UI 三处是否一致

## 运行习惯

- 代码修改后优先做编译校验
- 如果改动影响 UI、同步或代理流程，优先再做一次实际运行检查
- 遇到环境缺失、依赖缺失或运行阻塞时，先确认原因，再决定是否让用户创建环境
